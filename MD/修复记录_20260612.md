## 修复清单

### Fix 1: 进度数据不持久化 [CRITICAL]

**问题**：`POST /api/user/progress/read` 返回成功但数据不写入数据库

**根因**：TypeORM JSON 列脏检查失败。`updateSkillProgress()` 中 `const pathData = path.pathData || {}` 获取的是同一对象引用，原地修改后赋值回去，TypeORM 认为没有变化，跳过 `path_data` 列的 UPDATE SQL。

**文件**：`backend-ts/src/modules/progress/progress.controller.ts` 第 141、180 行

**修复**：对 `pathData` 做深拷贝，强制 TypeORM 检测到变化

```typescript
// 修复前
const pathData = path.pathData || {};

// 修复后
const pathData = JSON.parse(JSON.stringify(path.pathData || {}));
```

**验证**：`POST /api/user/progress/read` 后查询 `learning_plans_v3.path_data`，JavaScript 的 `read_at` 字段已更新。

### Fix 2: 进度更新不联动技能掌握度 [CRITICAL]

**问题**：标记已读/完成后，`user_skills_v3.mastery_pct` 始终为 0，匹配度不变

**根因**：`ProgressController` 只修改 `learning_plans_v3.path_data`，从未注入或调用 `SkillService`，两张表完全断联。

**文件**：
- `backend-ts/src/modules/progress/progress.controller.ts`（注入 + 联动逻辑）
- `backend-ts/src/modules/progress/progress.module.ts`（导入依赖模块）

**修复**：
1. 注入 `SkillService` 和 `NotificationService`
2. `read` 操作后调用 `skillService.updateMastery(userId, skill, +25)`
3. `quiz` 通过后调用 `skillService.updateMastery(userId, skill, +25)`
4. `complete` 后调用 `skillService.setMastery(userId, skill, 100)` + 发送通知

**验证**：JavaScript 从 `masteryPct=0.00` 变为 `25.00`。

### Fix 3: 讲义生成不保存到知识库 [CRITICAL]

**问题**：`POST /api/user/agents/lecture` 每次重新调用 LLM，生成内容不缓存，`GET /api/user/learning-paths/knowledge/:skill` 始终返回 `has_content: false`

**根因**：`AgentsController` 直接返回 LLM 输出，未调用 `KnowledgeBaseService.saveLecture()`。`KnowledgeBaseService` 有 `saveLecture()` 方法但从未被 Agent 流程调用。

**文件**：
- `backend-ts/src/modules/agents/agents.controller.ts`（注入 + 保存逻辑）
- `backend-ts/src/modules/agents/agents.module.ts`（导入 KnowledgeModule）

**修复**：在 `generateLecture`、`generateReading`、`generateCode` 三个方法中，LLM 生成成功后调用对应的保存方法：

```typescript
// lecture → saveLecture
await this.knowledgeBase.saveLecture(skillName, result.data, level);

// reading → saveQuiz
await this.knowledgeBase.saveQuiz(skillName, result.data.questions);

// code → saveCoding
await this.knowledgeBase.saveCoding(skillName, result.data.problems);
```

**验证**：生成 HTML/CSS 讲义后，`GET /api/user/learning-paths/knowledge/HTML%2FCSS` 返回 `has_content: true`。

### Fix 4: 今日任务永远为空 [CRITICAL]

**问题**：`GET /api/user/learning-tasks/today` 返回 `mainTasks: [], sideTasks: []`

**根因**：`TaskSchedulerService.generateTodayTasks()` 的分配逻辑要求 `mainUsed + estimatedMin <= mainMinutes`。用户每天 2 小时 → `mainMinutes = 96`，但模板中最小技能 `HTML/CSS` 需要 150 分钟，所有技能都超过预算，循环结束后 tasks 为空。

**文件**：`backend-ts/src/services/task-scheduler.service.ts` 第 258、275 行

**修复**：添加保底逻辑，确保至少分配一个任务（与 `PlannerAgentService.generateDailyTasks` 对齐）

```typescript
// 修复前
if (isMain && mainUsed + estimatedMin <= mainMinutes) {

// 修复后
if (isMain && (mainUsed + estimatedMin <= mainMinutes || mainUsed === 0)) {
```

**验证**：返回 `mainTasks: 1 (HTML/CSS 150min), sideTasks: 1 (Node.js 200min)`。

### Fix 5: 匹配度永远为 0 [CRITICAL]

**问题**：所有岗位匹配度为 0，用户已有的 JavaScript/TypeScript 被标记为"缺失"

**根因**：`MatchAgentService.calculateSkillMatch()` 第 213 行要求 `masteryPct >= 60` 才算匹配。自报技能 `masteryPct=0.00` 全部低于阈值，被归入 `missing` 数组。

**文件**：`backend-ts/src/services/match-agent.service.ts` 第 212-219 行

**修复**：改为分级匹配，区分"没有技能"和"有技能但掌握度低"

```typescript
// 修复前
if (userSkill && userSkill.masteryPct >= 60) {
  matched.push(skill.name);
  matchedWeight += weight * (userSkill.masteryPct / 100);
} else {
  missing.push(skill.name);
}

// 修复后
if (userSkill) {
  matched.push(skill.name);
  const effectivePct = Math.max(userSkill.masteryPct, 10); // 最低 10% 基础分
  matchedWeight += weight * (effectivePct / 100);
} else {
  missing.push(skill.name);
}
```

**验证**：匹配度从 0 变为 3.9，`matched: ['JavaScript', 'TypeScript']`，`missing: ['React', 'CSS']`。

### Fix 6: Resume 页面白屏 [CRASH]

**问题**：访问 `/user/resume` 页面直接白屏，控制台报 `ReferenceError: useNavigate is not defined`

**根因**：`Resume.tsx` 第 23 行使用 `const navigate = useNavigate()` 但未从 `react-router-dom` 导入 `useNavigate`。且 `navigate` 变量在组件中从未被使用（死代码）。

**文件**：`frontend/src/pages/user/Resume.tsx` 第 23 行

**修复**：直接删除未使用的代码行

```typescript
// 删除这行
const navigate = useNavigate();
```

**验证**：页面正常加载，不再白屏。

### Fix 7: 通知 API 返回 500 [HIGH]

**问题**：`GET /api/user/notifications` 和 `GET /api/user/notifications/unread-count` 返回 500

**根因**：双层问题
1. **Redis 空指针**：`NotificationService` 构造函数注入 `@Inject(REDIS_CLIENT) private redis: Redis`，Redis 不可用时 `this.redis` 为 `null`，所有 Redis 操作抛出 "Cannot read properties of null"
2. **数据库缺列**：`notifications_v3` 表缺少 `status` 列，TypeORM 实体继承 `BaseEntity` 定义了 `status` 列，但实际表没有该列

**文件**：
- `backend-ts/src/services/notification.service.ts`（Redis 空值保护 + 查询容错）
- 数据库 `notifications_v3` 表（ALTER TABLE 添加 `status` 列）

**修复**：
1. 所有 Redis 操作添加 `if (this.redis)` 保护
2. 所有带 `status: 1` 的查询添加 try-catch 容错，失败时回退到不带 `status` 的查询
3. 执行 `ALTER TABLE notifications_v3 ADD COLUMN status TINYINT DEFAULT 1`

**验证**：`GET /api/user/notifications` 返回 200，`GET /api/user/notifications/unread-count` 返回 `{count: 0}`。
