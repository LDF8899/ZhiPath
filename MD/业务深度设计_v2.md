## 一、产品定位修订

### 1.1 核心价值重定义

```
旧定位：面向学生的 AI 人岗匹配与学习路径系统
新定位：专业驱动的个性化学习 + 岗位匹配平台

核心逻辑：
├── 用户的专业 → 决定基础课程方向
├── 用户的目标岗位 → 决定主线学习任务
├── 用户的兴趣 → 决定支线学习内容
└── 最终目的 → 人岗匹配（技能达标即可投递）
```

### 1.2 与招聘平台的本质区别

```
Boss 直聘：用户来找工作 → 看岗位 → 投简历
智途：     用户来变强   → 学技能 → 匹配岗位 → 投简历

智途的核心竞争力：
不是"AI 推荐岗位"（Boss 也在做）
而是"AI 驱动的技能差距分析 + 个性化学习路径 + 学习过程验证"

学生来这里不是为了找工作，而是为了变得够格去找到工作。
```

## 二、学习计划系统

### 2.1 多计划架构

```
用户可以同时拥有多个学习计划：

计划 1：前端开发（主线）
├── 阶段1：HTML/CSS/JavaScript 基础
├── 阶段2：React + TypeScript
└── 阶段3：工程化 + 项目实战

计划 2：Python（支线/兴趣）
├── 阶段1：Python 基础
└── 阶段2：爬虫/数据分析

计划 3：考研（如有需要）
└── ...
```

### 2.2 计划与岗位的关系：解绑但联动

```
核心原则：
├── 计划是"学技能的路径"
├── 岗位是"求职的目标"
├── 两者独立，但通过"技能画像"联动

联动机制：
计划 → 累积技能 → 技能画像（MongoDB）
                        ↓
                岗位匹配用技能画像
                        ↓
                匹配产生"技能差距"
                        ↓
                差距 → 反哺学习建议
```

### 2.3 Git 分支模型

```
学习计划采用 Git 思想：

主计划（前端方向）
├── 学了 React
├── 学了 CSS
│
├── → 分支：用户想补 Docker（因为看了岗位要求）
│   └── 学了 Docker
│
├── → 分支：用户兴趣驱动想学 Python
│   └── 学了 Python
│
└── 所有分支的技能最终汇入技能画像
```

### 2.4 计划内部结构：时间驱动框架

```
用户输入约束：
├── "我有 90 天想达到初级前端水平"
│   → AI 反推：90天 ÷ 每个技能所需时间 = 学习计划
│
└── "我每天能学 2 小时"
    → AI 反推：2小时/天 × N天 = 总容量 → 安排技能

计划结构：
├── 总时长约束（天数 或 每日时长）
├── 分阶段（Phase）
├── 每阶段包含技能点
├── 每个技能点预估学习时长
└── 生成日程表（日历展示）
```

### 2.5 计划生命周期

```
新建计划：
├── 快速开始 → 简表单（最少信息，先跑起来）
└── 详细计划 → AI 对话（深度收集，写入个人中心）

切换计划：
├── 系统默认加载上次退出时的计划
├── 弹窗询问"继续上次计划？还是开启新计划？"
└── 新计划可导入旧计划的技能/进度（技能属于用户不属于计划）

计划调整：
├── 用户修改主线/支线比例 → 系统动态调整周期或时长
├── 弹窗提醒用户影响："计划周期 90天 → 120天"
└── 用户可选择"增加每日时长保持原周期"
```

## 三、用户进入流程（Onboarding 修订）

### 3.1 首次登录流程

```
用户注册/登录（首次）
    ↓
AI 浮窗自动弹出，主动打招呼：
"你好！我是智途 AI 助手，先了解一下你的情况吧"
    ↓
AI 通过对话收集关键信息：
├── 必须：专业、年级/毕业年份、意向方向
├── 重要：已有技能、每日可用学习时间
└── 可选：目标岗位、项目经历、GitHub
    ↓
信息足够 → 生成第一个学习计划
    ↓
进入主页，开始学习
```

### 3.2 非首次登录流程

```
用户登录
    ↓
弹窗打招呼：
"早上好！上次你在学 React Hooks，要继续吗？"
    ↓
├── [继续上次计划] → 加载数据，进入主页
├── [切换计划] → 显示计划列表
└── [开启新计划] → 可导入旧计划技能/进度
    ↓
进入主页
```

### 3.3 Onboarding 两种模式

```
快速开始：
├── 只收集最少信息（专业 + 方向 + 每日时长）
├── 弹出简表单
├── 立刻生成初始计划
└── 后续在个人中心补充

详细计划：
├── 与 AI 对话深度收集
├── AI 主动索取信息（技能、项目、偏好）
├── 对话内容自动写入个人中心
└── 生成更精准的计划
```

## 五、学习资源系统

### 5.1 多模态资源

```
每个技能点提供多种形式的学习资源：

📖 讲义      文字讲解 + 代码示例
🎬 视频      推荐教程链接
💻 编程题    Monaco Editor 在线练习
✏️ 练习题    选择题 / 填空题
🗺️ 图谱      知识图谱可视化

完成标准（多条件，可选）：
├── 读完讲义 → 基础完成
├── 做题通过 → 进阶完成
├── 编程题通过 → 深度完成
└── 阶段考试通过 → 确认掌握
```

### 5.2 资源生成机制

```
分层方案：计划创建时生成 + 全局公共缓存

用户创建计划（比如"前端开发"）
    ↓
AI 生成完整计划 → 列出所有技能点
    ↓
立刻提交异步任务：为这些技能点生成学习资源
    ↓
后台 Agent 队列处理（几分钟到几十分钟）
    ↓
资源逐个生成完毕 → 存入知识库（MongoDB）
    ↓
用户开始学习时 → 资源已就绪，秒开

公共知识库复用：
├── React Hooks → 已有（之前某个用户触发过）→ 直接用
├── Docker → 还没有 → 触发生成 → 存入公共库 → 下次复用
└── 知识库是全平台共享资产，不是 per 用户生成
```

### 5.3 资源质量保障

```
ReviewerAgent 审查流程：

LLM 生成内容
    ↓
自动审查（第一道关）：
├── 格式合规（JSON 结构）
├── 安全性过滤
├── 重复检测
└── 答案交叉验证（用另一个 LLM 验证）
    ↓
高置信度 → 直接使用
低置信度 → 管理员审核队列
    ↓
管理员审核（第二道关）：
├── 讲义准确性抽检（10%）
├── 题目答案审核
└── 简历发送前必须人工确认
```

## 六、技能模型

### 6.1 技能的四种来源与信任度

```
┌─────────────────────────────────────────────┐
│            Skill Sources                     │
│                                             │
│  1. Onboarding 自报    → 信任度: 低 (0.3)   │
│  2. 对话中提到         → 信任度: 中 (0.5)   │
│  3. GitHub 项目证明    → 信任度: 高 (0.7)   │
│  4. 考试通过           → 信任度: 确认 (1.0) │
│                                             │
│  匹配度计算时应加权：                         │
│  effective_skill = skill × trust_weight      │
└─────────────────────────────────────────────┘
```

### 6.2 技能掌握度量化（百分比）

```
百分比量化，便于画像和雷达图展示：

React Hooks — useState：
├── 读完讲义      → +30%
├── 选择题通过    → +25%
├── 编程题通过    → +25%
└── 阶段考试通过  → +20%
    └── 总计：100%

CSS Grid：
├── 读完讲义      → +30%
├── 选择题通过    → +25%
├── 编程题 未做    → +0%
└── 阶段考试 未考  → +0%
    └── 总计：55%
```

### 6.3 技能衰减模型

```
技能衰减曲线（简化版）：
- 考试通过后：确认状态，3个月内不衰减
- 3个月后未使用：每过1个月 trust_weight -0.1
- 重新做题/完成相关任务：恢复到 1.0

目的：防止"考完就忘"的技能充数
```

### 6.4 技能依赖图（Neo4j）

```
Neo4j 存技能-技能依赖关系：

React Hooks → depends_on → JavaScript ES6+
React Router → depends_on → React Hooks
Next.js → depends_on → React Router + Node.js 基础

PlannerAgent 生成路径时：
├── 按依赖拓扑排序
├── 先学基础，再学上层
└── 避免用户"看不懂"的挫败感
```

## 七、匹配度计算

### 7.1 匹配度公式（修订）

```
分场景权重（校招 vs 社招）：

校招（v1 主体）：
├── 必须技能覆盖度    30%（降低：校招生不太可能全掌握）
├── 加分技能覆盖度    15%
├── 项目经历相关度    15%
├── 考试成绩          20%（提高：考试证明学习能力）
├── 学习路径进度      10%（提高：学习态度很重要）
└── 学习速度          10%（新增：完成效率）

社招（v2）：
├── 必须技能覆盖度    40%
├── 加分技能覆盖度    20%
├── 项目经历相关度    25%
├── 考试成绩          10%
└── 学习路径进度      5%
```

### 7.2 匹配度可解释性

```
用户看到 65% 匹配度，需要知道怎么提高到 80%：

当前匹配度：65%
├── 必须技能：4/7 掌握 (57%) → 贡献 17.1/30
│   └── 缺少：TypeScript, Node.js, 数据库设计
├── 加分技能：2/5 掌握 (40%) → 贡献 6.0/15
├── 项目经历：有 React 项目 → 贡献 12.0/15
├── 考试成绩：React 85分 → 贡献 14.0/20
└── 学习进度：30% → 贡献 3.0/10

提升建议（按贡献度排序）：
1. 学习 TypeScript (+8% 预估)
2. 学习 Node.js (+5% 预估)
3. 完成当前阶段学习 (+3% 预估)
```

### 7.3 匹配度实时更新

```
匹配度应该是事件驱动的增量更新：

触发事件 → 匹配度变化：
├── 完成技能点学习     → +1~2%
├── 通过技能考试       → +3~5%
├── 通过阶段考试       → +10~15%
├── 添加项目经历       → +5~8%
└── 3天未学习          → -1%

展示方式：
├── Toast: "匹配度从 62% → 64% 🎉"
├── 弹窗: "恭喜！React Hooks 掌握认证通过"
└── 全屏庆祝动画（阶段考试通过时）
```

### 7.4 分阶段达标（岗位投递门槛）

```
不同岗位门槛不同：

初级岗位：掌握 60% 必须技能就能投递
├── 理由：校招生不需要全掌握
└── 展示："你已达标，可以投递！"

中级岗位：需要 80% + 阶段考试通过
└── 展示："再学 2 个技能就能达标"

高级岗位：需要 95% + 项目经历
└── 展示："距离达标还差较远，建议先学习"
```

## 八、岗位系统

### 8.1 岗位与计划的联动

```
用户在岗位页看到"全栈工程师"岗位，缺少 Docker 和 Redis：

系统提供三种操作（用户自选）：
├── A. 展示差距 + 推荐加入计划
│   "这个岗位你还差 Docker 和 Redis，要加入你的学习计划吗？"
│
├── B. 自动调整现有计划
│   "你选了这个岗位作为目标，我把你计划调整了一下，加了 Docker 和 Redis"
│
└── C. 创建新计划
    "这个岗位需要的技能和你现有计划有差距，要不要创建一个新计划专门补这些？"
```

### 8.2 岗位考试

```
岗位考试机制：

企业真题 + Agent 生成 → 管理员校准审核
    ↓
用户完成阶段学习后解锁
    ↓
├── 通过 → 匹配度大幅提升 → 可投递简历
└── 未通过 → 错题分析 → 针对性学习计划 → 限次重考

重考规则：
├── 一周内最多重考 3 次
├── 必须完成针对性学习计划后才能重考
└── ReviewerAgent 分析错题并生成补强计划
```

## 九、智能体办公室

### 9.1 架构

```
主智能体（大脑）：IntentRouter + 编排
    ↓
多个子智能体协作：
├── ProfileAgent    → 分析用户画像
├── PlannerAgent    → 生成学习路径
├── ResourceAgent   → 生成学习资源
├── MatchAgent      → 计算匹配度
├── ReviewerAgent   → 质量审查
├── ResumeAgent     → 生成简历
└── JDParserAgent   → 解析岗位 JD
```

### 9.2 办公室页面

```
用户可以进入"智能体办公室"查看 Agent 工作状态：

┌─────────────────────────────────────────┐
│  🏢 智能体办公室                          │
│                                         │
│  🟢 PlannerAgent    空闲                 │
│  🟡 ResourceAgent   正在工作 — 60%       │  ← 实时进度
│  🟡 MatchAgent      正在工作             │
│  🟢 ReviewerAgent   空闲                 │
│  🔴 ResumeAgent     未启用               │
│                                         │
│  任务队列：                               │
│  ⏳ 生成 TypeScript 讲义（排队中）         │
│  ⏳ 生成 CSS Grid 讲义（排队中）           │
└─────────────────────────────────────────┘
```

### 9.3 用户交互

```
用户在办公室可以：
├── 拖拽排序任务优先级
├── 标记任务为"紧急"（优先生成）
├── 跳过/取消不需要的任务
└── 查看已完成任务历史

双重价值：
├── 用户体验：不用干等，可以先学已生成的资源
└── 系统意图：用户标记"紧急" = 告诉系统"我接下来要学这个"
              → 系统优先生成 → 同时调整今日任务推荐
```

## 十、简历系统

### 10.1 多版本管理（Git 思想）

```
简历系统 Git 模型：

base（基础简历）
├── 包含用户所有信息（个人中心全部数据）
│
├── → v1-前端开发工程师
│   ├── 突出 React/TypeScript 项目
│   ├── 弱化后端经历
│   └── 用户编辑后导出 PDF
│
├── → v2-全栈工程师
│   ├── 前后端均衡展示
│   └── 用户编辑后导出 PDF
│
└── 用户更新了新项目经历
    → 可选择"合并到所有版本"或"只加到某个版本"
```

### 10.2 简历生成流程

```
ResumeAgent 生成简历：

1. 读取个人中心全部数据
   ├── 基本信息
   ├── 技能画像 + 掌握百分比
   ├── 项目经历（GitHub + 手动）
   ├── 考试成绩
   ├── 学习成果
   └── 自我评价

2. 根据目标岗位调整简历重点
   ├── 突出匹配的技能和经历
   └── 弱化不相关的部分

3. 用 HTML 生成初版简历

4. 用户在线编辑修改

5. 导出 PDF
   ├── A4 纸张大小
   ├── 排版优化，减少大片留白
   └── 关键信息不断页
```

```
用户退出之前，系统自动保存当天学习记录：

## 十二、考试机制修订

### 12.1 题目质量保障

```
LLM 出题 → 自动校验：
├── 用另一个 LLM 验证答案（交叉验证）
├── 检查干扰选项的区分度

考试后反馈循环：
├── 某题通过率 > 95% → 太简单，降低权重或替换
├── 某题通过率 < 20% → 太难 or 答案有误，人工审核
└── 某题被大量用户标记"答案有误" → 自动下架 + 告警
```

```
考试未通过
    ↓

├── 归类：哪些知识点错误
├── 归因：概念不清？还是粗心？
└── 生成：针对性学习计划（补强计划）

├── 只包含错题涉及的知识点
├── 推荐相关讲义 + 练习题
└── 预计完成时间：1-3天
    ↓
完成补强计划 → 解锁重考
```

目的：
- 快速给用户正反馈

## 十六、每日任务调度算法

### 16.1 任务抽取逻辑

```
每日任务不是随机生成，而是从学习路径中按日程表"抽取"：

输入：
├── 当前学习路径（path_data JSON）
├── 当前阶段（current_phase）
├── 今日日期
├── 用户每日可用时长
└── 主线/支线比例

算法：
1. 读取路径中"当前阶段"的技能列表
2. 过滤掉已完成的技能（掌握度 ≥ 80%）
3. 按依赖拓扑排序，取排在最前面的 N 个技能
4. N 的计算：可用时长 × 主线比例 ÷ 单个技能预估时长
5. 支线同理：从支线计划中取，用剩余时长

输出：
├── 主线任务列表（含预估时长）
├── 支线任务列表（含预估时长）
└── 总预估时长
```

### 16.2 动态调整策略

```
用户学习速度与预估不符时的调整：

场景1：用户学得快（实际用时 < 预估的 70%）
├── 当天自动追加下一个技能点
├── 更新后续日程（提前完成）
└── Toast: "你学得很快！今天多学一个知识点？"

场景2：用户学得慢（实际用时 > 预估的 150%）
├── 当天任务缩减，未完成的顺延到明天
├── 更新后续日程（延后）
└── 通知: "今天内容较多，已帮你调整到明天"

场景3：用户连续3天未学习
├── 发送提醒通知
├── 今日任务量自动减少 50%（降低回归门槛）
└── 推荐一个"温故知新"的短任务（10分钟）
```

### 16.3 任务完成状态机

```
任务状态流转：

PENDING（待开始）
    ↓ 用户点击"开始学习"
IN_PROGRESS（进行中）
    ↓ 用户完成讲义阅读
LECTURE_DONE（讲义完成，基础完成）
    ↓ 用户完成练习题且通过
PRACTICE_DONE（练习完成，进阶完成）
    ↓ 用户完成编程题且通过
CODE_DONE（编程完成，深度完成）
    ↓ 用户通过阶段考试
EXAM_DONE（考试通过，确认掌握）

任何状态都可以跳过后续步骤直接标记为 SKIPPED（跳过）
用户可以随时将任务标记为 DONE（手动完成）
```

│  ├── 新增技能：Node.js, Express, MongoDB (5个)│

│                                              │
│  [确认更换]  [取消]                            │
└──────────────────────────────────────────────┘
```

### 18.2 路径重算算法

```
PlannerAgent 执行增量更新：

1. 获取旧岗位的必须技能集合 A
2. 获取新岗位的必须技能集合 B
3. 获取用户已掌握的技能集合 U

计算差异：
├── 保留技能：A ∩ B（两个岗位都需要）
├── 归档技能：A - B（旧岗位需要，新岗位不需要）
│   └── 如果用户已掌握 → 归档到"已学技能"，不丢失
├── 新增技能：B - A（新岗位需要，旧岗位不需要）
│   └── 如果用户已掌握 → 直接标记完成
│   └── 如果未掌握 → 加入学习路径
└── 前置技能：新增技能的依赖链中，用户未掌握的

生成新路径：
├── 保留技能 → 保持当前进度
├── 已掌握的新技能 → 标记完成
├── 未掌握的新技能 → 按依赖拓扑排序 → 生成新阶段
└── 重新计算日程表
```

### 18.3 日程调整

```
日程调整策略：

情况1：新岗位要求更少
├── 路径缩短，预计达成日期提前
└── 通知: "新目标更简单，预计提前 2 周达成！"

情况2：新岗位要求更多
├── 路径延长，预计达成日期延后
├── 如果延后超过用户 deadline
│   └── 建议增加每日学习时长，或降低支线比例
└── 通知: "新目标需要更多技能，预计延长 3 周"

情况3：技能高度重叠
├── 路径变化很小
└── 通知: "两个岗位很相似，只需额外学 2 个技能"
```

### 19.2 关系类型

```cypher
// 技能 → 岗位：必须技能
(:Skill)-[:REQUIRED_FOR {weight: 0.8}]->(:Job)

// 技能 → 岗位：加分技能
(:Skill)-[:PREFERRED_FOR {weight: 0.3}]->(:Job)

// 技能 → 技能：前置依赖
(:Skill)-[:DEPENDS_ON]->(:Skill)
// 例：React Hooks -[:DEPENDS_ON]-> JavaScript ES6+

// 知识点 → 技能：归属
(:KnowledgePoint)-[:BELONGS_TO]->(:Skill)

// 知识点 → 知识点：前后关系
(:KnowledgePoint)-[:NEXT]->(:KnowledgePoint)

// 用户 → 技能：已掌握
(:User)-[:MASTERED {level: 85, verified: true}]->(:Skill)

// 用户 → 岗位：目标
(:User)-[:TARGET_JOB {set_at: timestamp()}]->(:Job)
```

### 19.3 常用查询

```cypher
// 查询岗位的所有必须技能
MATCH (s:Skill)-[r:REQUIRED_FOR]->(j:Job {id: $jobId})
RETURN s.name, r.weight
ORDER BY r.weight DESC

// 查询用户与目标岗位的技能差距
MATCH (u:User {id: $userId})-[:TARGET_JOB]->(j:Job)
MATCH (s:Skill)-[:REQUIRED_FOR]->(j)
OPTIONAL MATCH (u)-[m:MASTERED]->(s)
RETURN s.name,
       CASE WHEN m IS NOT NULL THEN m.level ELSE 0 END as user_level,
       CASE WHEN m IS NOT NULL THEN true ELSE false END as has_skill

// 查询技能的前置依赖链
MATCH path = (s:Skill {id: $skillId})-[:DEPENDS_ON*]->(prereq:Skill)
RETURN prereq.name, length(path) as depth
ORDER BY depth DESC

// 查询掌握某技能后可以匹配的所有岗位
MATCH (u:User {id: $userId})-[:MASTERED]->(s:Skill)
MATCH (s)-[:REQUIRED_FOR]->(j:Job)
WITH j, count(s) as matched_skills
MATCH (all_s:Skill)-[:REQUIRED_FOR]->(j)
WITH j, matched_skills, count(all_s) as total_skills
WHERE matched_skills >= total_skills * 0.6
RETURN j.title, j.company, matched_skills, total_skills,
       round(matched_skills * 100.0 / total_skills) as match_pct
ORDER BY match_pct DESC
```

### 20.2 SearXNG 采集流程

```typescript
// 任务编排示例
class TaskOrchestrator {
  // 串行执行：A 完成后再执行 B
  async sequential(tasks: AgentTask[]): Promise<any[]> {
    const results = [];
    for (const task of tasks) {
      const result = await this.executeTask(task);
      results.push(result);
    }
    return results;
  }

  // 并行执行：A 和 B 同时执行
  async parallel(tasks: AgentTask[]): Promise<any[]> {
    return Promise.all(tasks.map(t => this.executeTask(t)));
  }

  // 依赖执行：B 依赖 A 的结果
  async withDependency(
    taskA: AgentTask,
    taskBFactory: (resultA: any) => AgentTask
  ): Promise<any> {
    const resultA = await this.executeTask(taskA);
    const taskB = taskBFactory(resultA);
    return this.executeTask(taskB);
  }
}
```

class PuppeteerPool {

      args: ['--no-sandbox', '--disable-setuid-sandbox'],
    });

    return pdf;
  }

  async onModuleDestroy() {
    await this.browser.close();
  }
}

## 二十三、实时进度推送

### 23.1 技术方案：Server-Sent Events（SSE）

```
选择 SSE 而非 WebSocket 的理由：
├── 单向推送（服务端 → 客户端），符合进度通知场景
├── 基于 HTTP，无需额外协议升级
├── 浏览器原生支持（EventSource API）
├── 自动重连
└── 比 WebSocket 简单，NestJS 原生支持

适用场景：
├── 智能体办公室进度更新
├── 资源生成完成通知
├── 匹配度变化通知
└── 考试成绩出来通知
```

### 23.2 实现方案

```typescript
// 后端：SSE 端点
@Controller('api/user/events')
class EventsController {
  @Get('stream')
  @Sse()
  stream(@CurrentUser('sub') userId: number): Observable<MessageEvent> {
    return this.eventsService.getUserEvents(userId);
  }
}

// 事件服务
@Injectable()
class EventsService {
  private userStreams = new Map<number, Subject<MessageEvent>>();

  // Agent 更新进度时调用
  emitProgress(userId: number, data: ProgressEvent) {
    const stream = this.userStreams.get(userId);
    if (stream) {
      stream.next({ data } as MessageEvent);
    }
  }

  getUserEvents(userId: number): Observable<MessageEvent> {
    if (!this.userStreams.has(userId)) {
      this.userStreams.set(userId, new Subject());
    }
    return this.userStreams.get(userId).asObservable();
  }
}

// 前端：监听事件
const eventSource = new EventSource('/api/user/events/stream');
eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  switch (data.type) {
    case 'agent_progress':
      updateAgentProgress(data.agent, data.progress);
      break;
    case 'resource_ready':
      showNotification(`"${data.skill_name}" 资源已生成！`);
      break;
    case 'match_score_update':
      updateMatchScore(data.new_score);
      break;
  }
};
```

## 二十四、考试防作弊机制

### 24.1 V1 策略（基础防作弊）

```
防作弊措施：

1. 题库大于考试题数 → 随机抽题
   ├── 题库 100 题，每次考试随机抽 20 题
   └── 同一用户每次考试题目不同

2. 选项顺序随机
   └── 选择题的选项每次随机排列

3. 时间限制
   ├── 选择题：每题 60 秒
   ├── 填空题：每题 90 秒
   ├── 编程题：每题 15 分钟
   └── 总时间限制：题目数 × 平均时间 × 1.5

4. 编程题用测试用例验证
   ├── 不只看输出结果
   ├── 运行多个测试用例（包含边界情况）
   └── 隐藏测试用例（用户看不到）

5. 答题行为记录
   ├── 记录每题的答题时间
   ├── 异常检测：答题时间 < 3 秒 → 标记
   └── 异常检测：全对且时间极短 → 标记
```

### 24.2 V2 策略（增强防作弊）

```
增强措施：

1. 题目难度自适应
   ├── 答对 → 下一题更难
   ├── 答错 → 下一题更简单
   └── 防止"背答案"策略

2. 答案相似度检测
   ├── 编程题：检测代码相似度（与网上答案对比）
   └── 作文题：检测 AI 生成痕迹

3. 考试环境检测（可选）
   ├── 全屏模式
   ├── 禁止切换标签页
   └── 检测剪贴板操作
```

### 26.1 岗位管理流程
