## 第一章：平台定位

### 1.1 一句话定义

**智途是一个面向在校学生的、专业驱动的个性化学习与人岗匹配平台。**

### 1.2 核心逻辑

```
用户的专业 → 决定基础课程方向
用户的目标岗位 → 决定主线学习任务
用户的兴趣 → 决定支线学习内容
最终目的 → 人岗匹配（技能达标即可投递）
```

### 1.3 与招聘平台的本质区别

```
Boss 直聘：用户来找工作 → 看岗位 → 投简历
智途：     用户来变强   → 学技能 → 匹配岗位 → 投简历

智途的核心竞争力：
不是"AI 推荐岗位"（Boss 也在做）
而是"AI 驱动的技能差距分析 + 个性化学习路径 + 学习过程验证"

学生来这里不是为了找工作，而是为了变得够格去找到工作。
```

### 1.4 用户分层

| 层级 | 定义 | 匹配策略 |
|------|------|----------|
| 小白 | 未完成 Onboarding | 潜力匹配（提示"完善信息后匹配更准确"） |
| 有画像无简历 | 完成 Onboarding 无项目经历 | 宽泛匹配（提示"完善简历提升精准度"） |
| 有画像有简历 | 完成 Onboarding + 项目经历 | 精准匹配 |

### 1.5 版本边界

| 版本 | 主体 | 说明 |
|------|------|------|
| v1（当前） | 在校学生 | 学生身份为第一版主体 |
| v2（规划） | 职场人 | 第二版扩展，需重新设计画像模型 |

## 第二章：Agent 体系

### 2.1 架构：主智能体 + 子智能体协作

```
主智能体（大脑）：IntentRouter + 编排引擎
    ↓
多个子智能体协作：
├── ProfileAgent    → 分析/更新用户画像
├── PlannerAgent    → 生成/调整学习路径
├── ResourceAgent   → 生成学习资源（讲义/题目/编程题）
├── MatchAgent      → 计算人岗匹配度
├── ReviewerAgent   → 质量审查（所有 LLM 产出必须过审）
├── ResumeAgent     → 生成针对性简历
└── JDParserAgent   → 解析岗位 JD
```

### 2.2 Agent 清单

| Agent | 职责 | 触发时机 | 数据源 |
|-------|------|----------|--------|
| **ProfileAgent** | 生成/更新用户画像 | Onboarding、GitHub 导入、对话提取 | 用户输入、GitHub API、对话历史 |
| **MatchAgent** | 计算人岗匹配度 | 画像更新、新岗位录入、技能变化 | 用户画像 + 岗位 JD + Neo4j |
| **PlannerAgent** | 生成学习路径 | 用户选定目标岗位/创建计划 | 用户画像 + 目标岗位 + 知识库 + 时间约束 |
| **ReviewerAgent** | 质量审查 | 知识库生成、简历投递、题目生成 | LLM 输出 + 知识库标准 |
| **ResumeAgent** | 生成针对性简历 | 用户在岗位页触发 | 用户画像 + 项目经历 + 目标 JD |
| **JDParserAgent** | 解析岗位 JD | 管理员录入岗位 | JD 原文 |
| **ResourceAgent** | 生成学习资源 | 计划创建、用户学习时 | 技能点信息 + 知识库 |
| **IntentRouter** | 意图识别与路由 | 每次 AI 对话 | 用户消息 + 页面上下文 |

### 2.3 Agent 通信规则

```
┌─────────────┐
│  用户行为    │
└──────┬──────┘
       ↓
┌─────────────┐
│ IntentRouter │ ← 每次对话必经，判断意图分发到对应 Agent
└──────┬──────┘
       ↓
┌──────────────────────────────────────────────┐
│  ProfileAgent ← MatchAgent ← PlannerAgent    │
│       ↑              ↑             ↑          │
│  GitHub导入      新岗位录入    用户选岗位      │
└──────────────────────────────────────────────┘
       ↓
┌─────────────┐
│ ReviewerAgent │ ← 所有 LLM 产出必须过审
└─────────────┘
```

### 2.4 Agent 输出规范

**所有 Agent 输出必须是结构化 JSON**，禁止返回纯文本。

```json
// ✅ 正确
{ "project_name": "...", "tech_stack": ["React"], "highlights": ["..."] }

// ❌ 错误 — 纯文本无法被前端解析
"这个项目使用了 React 和 FastAPI 技术栈..."
```

### 2.5 Agent 返回必须设置 current_agent

**[为什么]** 不设置会导致循环递归超限。

```ts
// ✅ 正确
return { current_agent: 'ProfileAgent', ...result };

// ❌ 错误 — 缺少 current_agent，下一轮不知道谁在处理
return { ...result };
```

### 2.6 Agent 间通信协议

使用 BullMQ（基于 Redis）实现任务队列：

```
队列名：agent:tasks
任务状态：pending → active → completed / failed

任务数据结构：
{
  "task_id": "uuid",
  "agent": "ResourceAgent",       // 目标 Agent
  "action": "generate_lecture",   // 动作
  "user_id": "123",
  "priority": 5,                  // 优先级 1-10，10 最高
  "params": { ... },
  "status": "pending",
  "progress": 0                   // 0-100
}
```

编排模式：
```
串行：A 完成后再执行 B（路径生成 → 资源生成）
并行：A 和 B 同时执行（多个技能点资源同时生成）
依赖：B 依赖 A 的结果（资源生成依赖路径生成的技能列表）
```

## 第三章：数据流规范

### 3.1 数据所有权

| 数据 | 存储位置 | 所有者 | 生命周期 |
|------|----------|--------|----------|
| 用户账号 | MySQL `users_v3` | 系统 | 永久 |
| 学生信息 | MySQL `students_v3` | 系统 | 永久 |
| 用户技能 | MySQL `user_skills_v3` | 用户 | 永久，持续更新 |
| 学习计划 | MySQL `learning_plans_v3` | 用户 | 跟随目标岗位 |
| 学习任务 | MySQL `learning_tasks_v3` | 用户 | 跟随计划 |
| 学习会话 | MySQL `learning_sessions_v3` | 用户 | 永久（Git commit） |
| 岗位 JD | MySQL `job_positions_v3` | 管理员 | 永久 |
| 考试记录 | MySQL `exam_records_v3` | 用户 | 永久 |
| 题库 | MySQL `exam_questions_v3` | 系统 | 永久 |
| 简历 | MySQL `resumes_v3` | 用户 | 永久 |
| 投递记录 | MySQL `job_applications_v3` | 用户 | 永久 |
| 通知 | MySQL `notifications_v3` | 用户 | 90 天 |
| 企业 | MySQL `enterprises_v3` | 管理员 | 永久 |
| 资讯 | MySQL `news_v3` | 系统 | 按类型定期更新 |
| 系统配置 | MySQL `system_config_v3` | 管理员 | 永久 |
| 用户画像 | MongoDB `user_profiles` | 用户 | 持续更新 |
| 对话历史 | Redis（热）+ MongoDB（冷） | 用户 | 会话级 |
| 知识库 | MySQL `knowledge_base_v3` | 系统 | **全平台复用，非 per 用户** |
| 知识图谱 | Neo4j | 系统 | 永久，岗位录入时自动更新 |

### 3.2 数据流全景

```
用户输入（Onboarding / 对话 / GitHub）
    ↓
ProfileAgent → 更新 user_skills_v3 + MongoDB user_profiles
    ↓
MatchAgent ← 用户技能 + 岗位 JD + Neo4j
    ↓
匹配度 → 前端展示
    ↓
用户选择目标岗位 → 创建学习计划
    ↓
PlannerAgent → 生成路径 → learning_plans_v3 + learning_tasks_v3
    ↓
用户学习 → 考试 → 进度更新 → 匹配度提升
    ↓
投递简历 → ReviewerAgent → 管理员决策
```

### 3.3 知识库运作机制

```
LLM 生成讲义大纲
    ↓
存入 MySQL knowledge_base_v3（全平台复用）
    ↓
Agent 发散生成多模态资源：
├── 选择题 / 填空题 / 编程题 / 作文题
└── 知识图谱数据 → Neo4j → React Flow 渲染
    ↓
ReviewerAgent 质量审查过滤
    ↓
每周定期更新：
SearXNG 搜索最新资料 → 与知识库对比 → 标记待审查 → Agent 判断是否更新
```

**核心原则：知识库是全平台共享资产，不是 per 用户生成物。**

### 3.4 知识图谱实时更新机制

```
新岗位录入（任意来源）
    ↓
BullMQ 接收异步任务
    ↓
JDParserAgent 解析 JD
├── 提取技能节点
└── 建立技能-岗位关系
    ↓
Neo4j 自动更新
    ↓
现有用户匹配度重新计算
    ↓
相关用户收到通知："有新岗位匹配你"
```

## 第四章：学习计划系统

### 4.1 多计划架构

```
用户可以同时拥有多个学习计划：

计划 1：前端开发（主线）
├── 阶段1：HTML/CSS/JavaScript 基础
├── 阶段2：React + TypeScript
└── 阶段3：工程化 + 项目实战

计划 2：Python（支线/兴趣）
├── 阶段1：Python 基础
└── 阶段2：爬虫/数据分析

计划 3：考研（如有需要）
└── ...
```

### 4.2 计划与岗位的关系：解绑但联动

```
核心原则：
├── 计划是"学技能的路径"
├── 岗位是"求职的目标"
├── 两者独立，但通过"技能画像"联动

联动机制：
计划 → 累积技能 → 技能画像（user_skills_v3）
                        ↓
                岗位匹配用技能画像
                        ↓
                匹配产生"技能差距"
                        ↓
                差距 → 反哺学习建议
```

### 4.3 Git 分支模型

```
学习计划采用 Git 思想：

主计划（前端方向）
├── 学了 React
├── 学了 CSS
│
├── → 分支：用户想补 Docker（因为看了岗位要求）
│   └── 学了 Docker
│
├── → 分支：用户兴趣驱动想学 Python
│   └── 学了 Python
│
└── 所有分支的技能最终汇入技能画像
```

### 4.4 计划内部结构：时间驱动框架

```
用户输入约束：
├── "我有 90 天想达到初级前端水平"
│   → AI 反推：90天 ÷ 每个技能所需时间 = 学习计划
│
└── "我每天能学 2 小时"
    → AI 反推：2小时/天 × N天 = 总容量 → 安排技能

计划结构：
├── 总时长约束（天数 或 每日时长）
├── 分阶段（Phase）
├── 每阶段包含技能点
├── 每个技能点预估学习时长
└── 生成日程表（日历展示）
```

### 4.5 计划生命周期

```
新建计划：
├── 快速开始 → 简表单（最少信息，先跑起来）
└── 详细计划 → AI 对话（深度收集，写入个人中心）

切换计划：
├── 系统默认加载上次退出时的计划
├── 弹窗询问"继续上次计划？还是开启新计划？"
└── 新计划可导入旧计划的技能/进度（技能属于用户不属于计划）

计划调整：
├── 用户修改主线/支线比例 → 系统动态调整周期或时长
├── 弹窗提醒用户影响："计划周期 90天 → 120天"
└── 用户可选择"增加每日时长保持原周期"
```

### 4.6 主线/支线定义

```
主线 = 目标岗位要求的技能（必须学）
支线 = 个人兴趣（想学就学）

用户可自定义比例：
├── 主线 80% / 支线 20%
├── 主线 70% / 支线 30%
└── 主线 60% / 支线 40%

比例调整时系统动态计算：
├── 计划周期变化（90天 → 120天）
├── 或每日时长变化（2h → 2.7h）
└── 弹窗提醒用户影响，用户确认后生效
```

### 4.7 换目标岗位机制

```
用户更换目标岗位
    ↓
弹出确认框（展示影响分析）：
├── 保留技能：React, JavaScript, CSS (8个)
├── 归档技能：无
├── 新增技能：Node.js, Express, MongoDB (5个)
├── 预估额外时间：3 周
└── 匹配度变化：68% → 52%（暂时下降）
    ↓
PlannerAgent 执行增量更新：
├── 保留技能：A ∩ B（两个岗位都需要）→ 保持进度
├── 归档技能：A - B（旧岗位需要，新岗位不需要）→ 已掌握的归档到"已学技能"，不丢失
├── 新增技能：B - A（新岗位需要，旧岗位不需要）→ 已掌握的标记完成，未掌握的加入路径
└── 前置技能：新增技能的依赖链中，用户未掌握的
    ↓
学习路径平滑更新，不推倒重来
```

**核心原则：技能属于用户，不属于岗位。换岗不丢技能。**

### 4.8 路径生成规则

```
用户选定目标岗位 + 设置时间约束
    ↓
PlannerAgent 生成学习路径
├── 按技能依赖关系拓扑排序
├── 每个阶段包含 3-5 个技能点
├── 每个技能点预估学习时长
├── 考虑用户每日可用时长
└── 生成日程表
```

### 4.9 今日任务优先级

```
主线任务（岗位驱动）
└── 每天至少完成一个
    完成后 ↓
支线任务（自选学习）
└── 主线完成后解锁当天支线

总时长控制：
└── 不超过用户设定的每日学习时长
    岗位任务优先填满时间
```

## 第五章：用户进入流程

### 5.1 首次登录流程

```
用户注册/登录（首次）
    ↓
AI 浮窗自动弹出，主动打招呼：
"你好！我是智途 AI 助手，先了解一下你的情况吧"
    ↓
AI 通过对话收集关键信息：
├── 必须：专业、年级/毕业年份、意向方向
├── 重要：已有技能、每日可用学习时间
└── 可选：目标岗位、项目经历、GitHub
    ↓
信息足够 → 生成第一个学习计划
    ↓
进入主页，开始学习
```

### 5.2 非首次登录流程

```
用户登录
    ↓
弹窗打招呼：
"早上好！上次你在学 React Hooks，要继续吗？"
    ↓
├── [继续上次计划] → 加载数据，进入主页
├── [切换计划] → 显示计划列表
└── [开启新计划] → 可导入旧计划技能/进度
    ↓
进入主页
```

### 5.3 Onboarding 两种模式

```
快速开始：
├── 只收集最少信息（专业 + 方向 + 每日时长）
├── 弹出简表单
├── 立刻生成初始计划
└── 后续在个人中心补充

详细计划：
├── 与 AI 对话深度收集
├── AI 主动索取信息（技能、项目、偏好）
├── 对话内容自动写入个人中心
└── 生成更精准的计划
```

### 5.4 5分钟速测（冷启动）

```
场景：用户刚完成 Onboarding，还没有任何技能确认

速测流程：
1. 根据用户选择的"意向方向"，推送 5 道基础题
2. 即时评分 + 技能确认
3. "你已经掌握了 JavaScript 基础！匹配度 +10%"

目的：
- 快速给用户正反馈
- 把"自报技能"升级为"测试确认技能"
- 建立"做题 → 进步"的心智模型
```

## 第七章：学习资源系统

### 7.1 多模态资源

```
每个技能点提供多种形式的学习资源：

📖 讲义      文字讲解 + 代码示例
🎬 视频      推荐教程链接
💻 编程题    Monaco Editor 在线练习
✏️ 练习题    选择题 / 填空题
🗺️ 图谱      知识图谱可视化

完成标准（多条件，可选）：
├── 读完讲义 → 基础完成（+30%）
├── 做题通过 → 进阶完成（+25%）
├── 编程题通过 → 深度完成（+25%）
└── 阶段考试通过 → 确认掌握（+20%）
```

### 7.2 资源生成机制

```
分层方案：计划创建时生成 + 全局公共缓存

用户创建计划（比如"前端开发"）
    ↓
AI 生成完整计划 → 列出所有技能点
    ↓
立刻提交异步任务：为这些技能点生成学习资源
    ↓
后台 Agent 队列处理（几分钟到几十分钟）
    ↓
资源逐个生成完毕 → 存入知识库（MySQL knowledge_base_v3）
    ↓
用户开始学习时 → 资源已就绪，秒开

## 第十四章：考试机制

### 14.1 考试类型

| 类型 | 触发条件 | 题目来源 | 通过效果 |
|------|----------|----------|----------|
| 通用技能考试 | 完成该技能点学习 | Agent 出题 → 管理员审核 | 知识点完成 → 路径进度推进 |
| 岗位考试 | 完成阶段学习后解锁 | 企业真题 + Agent 生成 → 管理员校准 | 匹配度大幅提升 → 可投递简历 |
| 5分钟速测 | Onboarding 完成后 | Agent 出题（5道基础题） | 快速确认技能 + 匹配度提升 |

### 14.2 考试流程

```
用户完成技能点学习
    ↓
参加通用技能考试
    ├── 通过 → 知识点完成 → 路径进度推进
    └── 未通过 → ReviewerAgent 分析错题
                → 自动生成针对性学习计划（补强计划）
                → 一周内限次重考
    ↓
阶段全部完成
    ↓
解锁岗位考试
    ├── 通过 → 匹配度大幅提升 → 可投递简历
    └── 未通过 → 同上
```

### 14.3 重考规则

- **限次**：一周内最多重考 3 次
- **错题分析**：每次未通过后，ReviewerAgent 必须分析错题并生成针对性学习计划
- **不允许裸考**：必须完成针对性学习计划后才能重考

### 14.4 题目质量保障

```
LLM 出题 → 自动校验：
├── 用另一个 LLM 验证答案（交叉验证）
├── 检查干扰选项的区分度
├── 检查与已有题目的相似度（去重）
└── 标记 confidence_score

高置信度（≥0.9）→ 直接入题库
低置信度（<0.7）→ 管理员审核队列

考试后反馈循环：
├── 某题通过率 > 95% → 太简单，降低权重或替换
├── 某题通过率 < 20% → 太难 or 答案有误，人工审核
└── 某题被大量用户标记"答案有误" → 自动下架 + 告警
```

### 14.5 防作弊机制

```
V1 策略（基础防作弊）：

1. 题库大于考试题数 → 随机抽题（同一用户每次题目不同）
2. 选项顺序随机
3. 时间限制（选择题60秒/填空题90秒/编程题15分钟）
4. 编程题用测试用例验证（含隐藏测试用例）
5. 答题行为记录（答题时间 < 3秒 → 标记异常）

V2 策略（增强防作弊）：
1. 题目难度自适应（答对→更难，答错→更简单）
2. 答案相似度检测（代码/作文）
3. 考试环境检测（全屏/禁止切换标签页，可选）
```

## 第十七章：AI 浮窗与上下文感知

### 17.1 上下文感知规则

AI 浮窗的行为由当前页面决定，**同一用户在不同页面看到不同的 AI 提示词和行为**。

| 页面 | AI 角色 | session 隔离 | 核心职责 |
|------|---------|-------------|----------|
| `/user/home` | 今日任务助手 | 独立 session | 引导开始今日任务 |
| `/user/learning`（岗位 Tab） | 学习进度助手 | 独立 session | 感知当前阶段和技能点 |
| `/user/learning`（自选 Tab） | 学习推荐助手 | 独立 session | 澄清需求，推荐方向 |
| `/user/jobs` | 岗位分析助手 | 独立 session | 分析技能差距，建议加入路径 |
| `/user/profile` | 画像完善助手 | 独立 session | 提示未填字段 |
| `/user/news` | 阅读理解助手 | **临时 session** | 解释名词，总结内容，不影响画像 |

### 17.2 三层记忆架构

```
Layer 1: Global Profile (MongoDB user_profiles)
├── 用户画像（所有页面共享）
├── 技能列表
└── 目标岗位
→ 所有页面可读写

Layer 2: Page Session (Redis)
├── 当前页面的对话上下文
├── 页面特定状态
└── 20条消息滑动窗口
→ 仅当前页面可读写

Layer 3: Temporary (内存)
├── 资讯页的临时对话
└── 不持久化，关闭即消失
→ 仅当前页面可用

跨页面传递机制：
AI 在学习页发现用户"对算法很感兴趣"
→ 写入 Layer 1 (profile.chat_insights)
→ 岗位页的 AI 可以读取并推荐算法岗
```

### 17.3 资讯页特殊规则

- 对话是临时 session，**不存入 MongoDB**
- **不影响用户画像**（阅读行业资讯不等于掌握该技能）

### 19.3 岗位管理流程

```
管理员录入岗位：

1. 填写基本信息（岗位名称/公司/薪资/地点/级别）
2. 粘贴 JD 原文（或手动填写）
3. 触发 JDParserAgent 自动解析
   ├── 提取必须技能列表
   ├── 提取加分技能列表
   └── 标记置信度
4. 管理员审核解析结果（确认/修改技能列表，设置投递门槛）
5. 上架岗位
   ├── 更新 Neo4j 知识图谱
   ├── 触发所有用户匹配度重新计算
   └── 匹配度 ≥ 60% 的用户收到通知
```

### 19.4 题库审核流程

```
Agent 出题 → 管理员审核：

1. Agent 生成题目（批量），每题标记 confidence_score
2. 自动分流：
   ├── confidence ≥ 0.9 → 直接入题库（免审核）
   ├── confidence 0.7-0.9 → 管理员抽检（10%）
   └── confidence < 0.7 → 必须管理员审核
3. 管理员审核界面：[通过] / [修改后通过] / [拒绝重新生成]
4. 题目质量监控：通过率异常 → 自动标记
```

### 19.5 简历审核流程

```
简历投递处理：

1. 用户在岗位页生成简历 → 一键投递
2. ReviewerAgent 自动分析（匹配度/技能匹配/建议）
3. 管理员审核界面：
   ├── [通过] → 发送至企业联络员邮箱（Nodemailer）
   ├── [拒绝] → 填写拒绝原因 → 通知用户
   └── [待定] → 标记为待定
```

### 19.6 管理端看板数据

```
/dashboard 展示的指标：

用户数据：总用户数/本周新增/活跃用户/Onboarding完成率/增长趋势
学习数据：总学习时长/平均每日时长/完成技能点数/考试通过率/热门技能Top10

## 第二十章：技术规范

### 20.1 命名规范（最高优先级）

| 场景 | 规范 | 示例 |
|------|------|------|
| 后端返回 JSON | **snake_case** | `target_job`, `today_tasks` |
| 数据库 Entity 属性 | camelCase | `targetJobId` |
| 数据库列名 | snake_case | `target_job_id` |
| 前端类型字段 | **必须与后端 JSON 一致** | `target_job`（不是 `targetJob`） |
| JWT payload | `{ sub, username, role }` | `user.sub`（不是 `user.id`） |

### 20.2 API 响应信封

```json
{ "code": 200, "message": "success", "data": { ... } }
```

所有 Controller 必须用 `success()` / `pageSuccess()` 包装。

### 20.3 认证规范

- token 存 sessionStorage（关闭浏览器需重新登录）
- 401 时清除 token 并跳转登录页
- JWT payload 中用户 ID 字段是 `sub`

### 20.4 环境配置

```env
MONGODB_URL=mongodb://root:root@127.0.0.1:27017?authSource=admin
```

**MongoDB 必须加 `?authSource=admin`**，否则 500。

### 20.5 构建规范

- 改了 `.ts` 后必须 `npm run build`
- 开发时建议 `npm run start:dev`（自动重编译）
- 前端 `npm run dev` 自带 HMR，无需手动操作

## 第二十一章：禁止事项

### 绝对禁止

| 禁止 | 原因 |
|------|------|
| Agent 返回纯文本 | 前端无法解析，导致白屏 |
| Controller 用 `user.id` | JWT 里是 `user.sub`，`user.id` 是 undefined → 500 |
| 前端类型字段名与后端不一致 | 运行时访问 undefined → 白屏 |
| 知识库 per 用户生成 | 浪费资源，应全平台复用 |
| Agent 自行决策（不经管理员） | 决策权必须在人 |
| `localStorage` 存 token | 安全隐患，关闭浏览器不清除 |
| 改 .ts 不 build | 跑旧代码，改了等于没改 |

### 不推荐

| 不推荐 | 推荐替代 |
|--------|----------|
| 手动构造 API 响应 | 用 `success()` / `pageSuccess()` |
| 前端写 JSON 转换层 | 类型直接对齐后端 |
| 每个页面重复 loading/error 逻辑 | 抽取 `useFetch` hook |
| 无痕模式外调试 | 浏览器扩展会干扰控制台 |

## 附录 D：技术实现方案

### D.1 每日任务调度算法

```
每日任务从学习路径中按日程表"抽取"：

输入：当前路径 + 当前阶段 + 今日日期 + 每日可用时长 + 主线/支线比例

算法：
1. 读取路径中"当前阶段"的技能列表
2. 过滤掉已完成的技能（掌握度 ≥ 80%）
3. 按依赖拓扑排序，取排在最前面的 N 个技能
4. N = 可用时长 × 主线比例 ÷ 单个技能预估时长
5. 支线同理：从支线计划中取，用剩余时长

动态调整：
├── 用户学得快（实际 < 预估70%）→ 自动追加下一个技能点
├── 用户学得慢（实际 > 预估150%）→ 未完成的顺延到明天
└── 连续3天未学习 → 任务量减少50% + 推荐短任务（10分钟）
```

### D.2 学习进度持久化

```
热数据（Redis）：
├── 当前正在学习的技能点 + 阅读位置
├── 今日学习时长（实时累计）
├── 今日任务完成状态
└── TTL: 24h，过期后归档到 MongoDB

温数据（MongoDB learning_sessions 集合）：
├── 每日学习会话记录
├── 每个技能点的学习进度
└── 保留 90 天

冷数据（MySQL learning_sessions_v3）：
├── 会话摘要
└── 永久保留

进度恢复：Redis → MongoDB → MySQL 逐级检查
```

### D.3 知识图谱 Schema（Neo4j）

```cypher
// 节点类型
(:Skill {id, name, category, difficulty, avg_learning_hours, demand_score})
(:Job {id, title, company, level, salary_range, location, status})
(:KnowledgePoint {id, name, skill_id, difficulty, content_types})
(:User {id, name})

// 关系类型
(:Skill)-[:REQUIRED_FOR {weight}]->(:Job)
(:Skill)-[:PREFERRED_FOR {weight}]->(:Job)
(:Skill)-[:DEPENDS_ON]->(:Skill)
(:KnowledgePoint)-[:BELONGS_TO]->(:Skill)
(:User)-[:MASTERED {level, verified}]->(:Skill)
(:User)-[:TARGET_JOB {set_at}]->(:Job)
```

### D.4 资讯系统数据流

```
三种来源：
1. 行业动态（SearXNG 每天2次自动抓取）
2. 技术趋势（Agent 每周生成 + ReviewerAgent 审核）
3. 招聘信息（管理员手动发布）

个性化推荐：
├── 基于用户技能画像（60%）
├── 基于目标岗位（30%）
├── 热门资讯兜底（10%）
└── 已读过滤（Redis SET 记录）
```

### D.5 简历 PDF 生成

```
技术方案：Puppeteer（headless Chrome）

流程：
1. 前端发送简历 HTML 到后端
2. Puppeteer 加载 HTML + 注入打印优化 CSS
3. 生成 A4 PDF（break-inside: avoid 防断页）
4. 上传到 MinIO → 返回下载链接

Puppeteer 连接池：复用 Chrome 实例，避免每次启动
```

### D.6 实时进度推送

```
技术方案：Server-Sent Events（SSE）

理由：单向推送、基于HTTP、浏览器原生支持、自动重连

事件类型：
├── agent_progress    — Agent 任务进度更新
├── agent_status      — Agent 状态变化（idle/working/error）
├── resource_ready    — 学习资源生成完成
├── match_score_update — 匹配度变化
├── exam_result       — 考试成绩出来
└── notification      — 通用通知
```
